  2. Three reasons D3 was helpful:
    1. D3 allowed for very easy dynamic charts simply by calling the same chart function again with slightly different data.
    2. D3's built in transitions/animations allowed for fluid and responsive visualizations that greatly help UX.
    3. D3's tracking and features allowed for easy tooltips, dynamically generated based on the data the user is hovering over.
  3. Three reasons D3 would not be the best tool for creating a visualization:
    1. D3 runs in Javascript, which requires the use of a browser. This is a non-trivial amount of processing/resources for a system, and there are many situations in which running a full browser engine is not feasible. For example, giving a presentation on a screen powered by an embedded machine that does not have a browser.
    2. D3 would not be a good choice to display simple, static data, as it requires a non-trivial amount of overhead (in resources, developer time, and network bandwidth) simply to set up. If a user wishes to simply display some basic static data, it makes more sense to generate a graph quickly in Excel or similar, and share that around. Using D3 for such a task would be a waste of developer time and computing power, where a simply .jpg export of an Excel graph would suffice.
    3. For advanced visualizations with extreme amounts of data, D3 would not be a good choice. D3 requires the client to load the data browser-side and do computations before finally rendering in the browser. This is not a feasible solution if the data were of medium to large size — most machines could not possibly load a few GBs (or TBs) of data into memory to process and render in the browser. For this use case, it makes much more sense to have the data in a dedicated database, and use a native application to process and display the data to the client.
